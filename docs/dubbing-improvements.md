# Идеи улучшения дубляжа — backlog

Deferred-инициативы, обсуждённые, но не реализованные. Каждая — самостоятельная история, можно браться в любом порядке. Внутри секций — субъективный приоритет по убыванию ожидаемой пользы.

## Как читать оценки

В каждом пункте две приближённые цифры по 100-балльной шкале:

- **Сложность реализации** — сколько работы и риска: 0 — несколько строк, никаких новых зависимостей, понятная архитектура; 100 — новая модель/GPU/лицензия/большой рефакторинг.
- **Прирост качества** — субъективный ожидаемый эффект на финальный дубляж (звук, синхрон, естественность, точность перевода — всё вместе): 0 — UX/уборка/защитный фолбэк; 100 — радикальное изменение восприятия.

Шкалы не калиброваны и предназначены для быстрой сортировки внутри backlog'а, не для абсолютного сравнения «X в N раз лучше Y».

## Уже сделано (для контекста)

- **`src/transcribe.py`** — `word_timestamps=True` + кастомная ресегментация по `.!?` → паузе ≥0.4с → лимиту 8с. Один сегмент = одна фраза.
- **`src/merge.py`** — эффективное окно `next.start − this.start` (использует естественные паузы перед следующим сегментом). `ATEMPO_MAX` 1.25 → 1.5. Последний сегмент проигрывается без ограничений.
- Реальный замер на 25-минутной лекции: `109/259` стрейчей → `54/259`, обрезано суммарно `8.40s` → `0.00s`.

## Качество перевода

### 1. Anti-hallucination guard в translate.py — _позаимствовать из vt-codex_

**Сложность: 15/100 · Эффект: 30/100**

**Проблема.** `translate.py` подаёт sliding-window историю N=3. LLM иногда «утекает» в контекст и возвращает перевод, который ближе к соседнему сегменту (предыдущему в истории), чем к текущему. Сейчас ловим только cyrillic-fallback и пустой ответ; смысловой leak тихий.

**Идея.** После `_clean(raw)` посчитать пересечение токенов:
- `current_overlap` — кандидат vs `seg.original` (текущий)
- `neighbor_overlap` — кандидат vs `prev.original` и/или `next.original`

Если `max(neighbor_overlap) >= 2 AND > current_overlap` → это leak, fallback на оригинал (как сейчас при не-кириллическом ответе).

**Файлы.** `src/translate.py` (~20 строк), тесты в `tests/test_translate.py`.

**Референс.** `vt-codex/src/vt_dub/ollama_rewrite.py`: `_token_set`, `_should_fallback_to_draft`. Они применяли это к stage rewrite, у нас единственный LLM-проход — там же и применять.

**Тонкость.** Подобие токенов на коротких сегментах ("Yes." vs "Yes, sure.") тоже даст overlap. Калибровать порог; возможно требовать `neighbor_overlap >= 3`.

**DoD.** Тест с подложным «leaked» response → ожидаем fallback. Запуск на боевом видео → проверить, что обычные переводы не триггерят guard.

### 2. Compaction-rewrite — второй LLM-проход

**Сложность: 25/100 · Эффект: 70/100**

**Проблема.** Русский TTS systematically на 30-50% длиннее английского. Текущий length hint «close to N characters ±20%» gemma часто игнорирует. Это и есть основной источник нагрузки на atempo.

**Идея.** Если `len(translated) > len(original) * 1.3` — второй проход с компактным промптом: «Сократи до X символов без потери смысла. Только русский, без объяснений.» Не рекурсивный вызов `translate` (попадёт в ту же ловушку с историей), а отдельная функция.

**Файлы.** Новая функция `compact()` в `src/translate.py` или `src/compact.py` + интеграция в pipeline (`src/dub.py`).

**Тонкость.** Удваивает Ollama-вызовы для длинных переводов. Применять только при overshoot (≈30-50% сегментов).

**DoD.** Замер: средний `len(translated) / len(original)` до и после. Цель — снизить с ≈1.4 до ≈1.15.

### 3. Few-shot примеры в system prompt

**Сложность: 5/100 · Эффект: 30/100**

**Идея.** В `SYSTEM_PROMPT` (`src/translate.py`) добавить 2-3 пары EN→короткий разговорный RU. Gemma с примерами стабильнее по стилю.

```
Examples:
EN: "Hi everyone, my name is Hannah."
RU: "Всем привет, меня зовут Ханна."

EN: "Today we're going to walk through prompting best practices."
RU: "Разберём лучшие практики промптинга."
```

**Файлы.** `src/translate.py`.

**DoD.** A/B на 1-2 минутах видео с/без примеров — оценить субъективно.

### 4. Жёсткий retry на overshoot (альтернатива #2)

**Сложность: 15/100 · Эффект: 50/100**

Дешевле compaction-pass, но менее надёжно. Если первый ответ длиннее `1.3 × len(original)` — повторить тот же запрос, добавив в system: `"STRICT: response MUST be <= N characters. Drop filler if needed."` Максимум 1 retry.

Брать, если #2 покажется тяжеловесом.

## Тайминг и сборка

### 5. Cascading playback в merge.py

**Сложность: 30/100 · Эффект: 35/100**

**Проблема.** Даже с эффективным окном, если у предыдущего сегмента TTS закончился рано, остаточная тишина «пропадает» — следующий сегмент жёстко стартует на `seg.start`.

**Идея.** Сместить старт: `effective_start = max(prev_audio_end, seg.start - max_shift)` где `max_shift` ≈ 1-2 сек, чтобы не уплыть от видео. TTS играет впритык, используя тишину предыдущего как фору.

**Файлы.** `src/merge.py` — переписать цикл с трекингом `prev_audio_end` (с учётом atempo). Тесты.

**Тонкость.** Меняет синхронизацию голос/видео. На лекциях с фронтальной камерой OK, на крупном плане губ — нет. Сделать `cascading_max_shift` в `config.yaml`, по умолчанию 0 (выключено).

**DoD.** На длинных сегментах с большой паузой — старт следующего смещается вперёд. Тест: два сегмента, у первого audio < window, у второго audio > window → второй начинает раньше своего start.

### 6. Удалить или починить group.py

**Сложность: 10/100 · Эффект: 5/100**

**Проблема.** После transcribe-фикса все паузы между сегментами ≥0.4с, а `gap_threshold=0.3` → `group_segments` практически никогда не сливает. Это мёртвый код.

**Варианты:**
- (a) **Удалить** — `group_segments` больше не нужен, ресегментер уже даёт правильные единицы.
- (b) **Перенацелить** — слипать «огрызки» (`duration < 1.5s`) с соседом через паузу ≤0.6с, чтобы не плодить микро-сегменты на быстрой речи.

**Файлы.** `src/group.py`, `src/dub.py`, `config.yaml`, тесты.

**DoD.** На реальном видео — кол-во сегментов длительностью <1.5с после group близко к нулю. Или: удалить файл и тесты, обновить `dub.py`.

## Альтернативные архитектуры (большие истории)

### 7. VAD-сегментация вместо word_timestamps

**Сложность: 30/100 · Эффект: 25/100**

**Идея.** `vad_filter=True` + tuned `VadOptions(min_silence_duration_ms=300-500, max_speech_duration_s=8)`. Silero-VAD режет по реальным паузам тишины — может быть надёжнее на быстрой речи без пунктуации.

**Когда брать.** Если на конкретных видео текущий resegmenter мажется (длинные монологи без точек, неестественные паузы Whisper-таймстемпов).

**Файлы.** `src/transcribe.py`. Не выкидывать `_resegment` — оставить как опцию, переключатель в config.

**Тонкость.** VAD не понимает пунктуации, может разрезать предложение по дыханию. Возможно — комбо: VAD-pre + наш resegment поверх.

**DoD.** A/B на 2-3 разных видео (лекция, диалог, монолог).

### 8. Two-stage Marian + Ollama — _полный паттерн vt-codex_

**Сложность: 60/100 · Эффект: 60/100**

**Идея.** Буквальный en→ru через `Helsinki-NLP/opus-mt-en-ru` (transformers), затем Ollama-polish для разговорной формулировки.

**За.** Marian не галлюцинирует, не путается в контексте, faithful. LLM отвечает только за стиль.

**Против.**
- ~300 MB зависимостей (`transformers`, `sentencepiece`, Marian-модель).
- Стек перестаёт быть Ollama-only.
- Rewrite-стадия требует guard'а из #1.

**Когда брать.** Только если #1+#2+#3 не дали стабильного качества. Тяжёлая артиллерия.

**Референс.** `vt-codex/src/vt_dub/translate.py` (Marian) и `vt-codex/src/vt_dub/ollama_rewrite.py` (rewrite stage).

### 9. Per-segment WAV + concat вместо amix

**Сложность: 35/100 · Эффект: 15/100**

**Текущее.** Все сегменты в один ffmpeg-граф через `adelay`+`amix`.

**Альтернатива.** Каждый сегмент → отдельный WAV → silence-fillers → concat manifest. Контроль точнее (можно дампать промежуточные артефакты), debug проще.

**За.** Дебажный артефакт `segments/0001.wav`, … Облегчает диагностику конкретного сбоя.

**Против.** Больше I/O, больше файлов в /tmp.

**Когда брать.** Если будут проблемы с amix-нормализацией громкости или нужен per-segment контроль.

**Референс.** `vt-codex/src/vt_dub/pipeline.py` (concat manifest), `vt-codex/src/vt_dub/timing.py`.

## Bonus: TTS

### 10. Замена Silero на XTTS/F5-TTS — voice cloning

**Сложность: 70/100 · Эффект: 90/100**

Silero — фиксированные 5 голосов. F5-TTS / XTTS поддерживают клонирование из 10-30 сек семпла оригинала. Радикально меняет восприятие — голос русской дорожки совпадает с оригиналом.

**Файлы.** Новый `src/tts_xtts.py`, переключатель в `config.yaml`.

**Когда брать.** Когда стандартные голоса Silero не устраивают / нужен «голос Ханны на русском».

**Тонкость.** XTTS требует GPU для разумной скорости. F5-TTS — CPU OK, но медленнее.

**Pre-synthesis speed estimation как первичный механизм синхронизации (только для XTTS/F5).** XTTS v2 принимает параметр `speed` в `tts.tts_to_file(speed=...)` — модель синтезирует сразу с нужной скоростью, без post-stretch. Это качественно лучше любого post-processing'а: голос изначально звучит ускоренно (как если бы человек реально говорил быстрее), без фазовых артефактов rubberband и без «булькания» atempo. AutoDub (`tools/audio_synthesis.py:202-218`) делает так: оценивает `speed = estimated_text_duration / target_window` (с эмпирической константой ~2.5 слов/сек — для русского нужно перекалибровать), кламп `[0.75, 3.0]`, передаёт в синтез. **Это меняет роль rubberband (#22)** при переходе на XTTS: rubberband становится fallback'ом для случаев, когда даже max speed не влез, а primary-механизм синхронизации переезжает внутрь TTS-движка. У Silero такого параметра нет — пометка релевантна только для XTTS/F5 веток.

### A. Улучшения текущих фич

#### 11. Trim trailing silence после TTS — _open-dubbing/text_to_speech.py:158-187_

> **Status (2026-05-03):** Проверено эмпирически на 3 видео из `data/`, ветка `experiment/trim-silence-benchmark`, отчёт — `experiments/results/2026-05-02-trim-benchmark.md`. Реальный эффект **≪ обещанного DoD**: `Σ audio` сократилось на 1.5–1.9% (vs 5–15%), `truncated_total` спасает ~0.8 с речи на 31-минутном видео, `overflow_count` снижается на 3–5%. На слух различия между none / silero / pydub / ffmpeg / numpy неощутимы. **Эффект понижен с 35/100 до ~5–10/100**, в `src/tts.py` не интегрировать. Pairwise sub-вывод: numpy / pydub / ffmpeg попарно ~5 мс расхождения (взаимозаменяемы); silero VAD — outlier (режет в ~10× меньше). Инфра эксперимента остаётся на ветке как каркас для повторной проверки при смене TTS (#10) или появлении vocal separation (#13).

**Сложность: 15/100 · Эффект: ~~35~~ → ~5/100** (после проверки)

**Проблема.** Silero часто добавляет хвостовое молчание к синтезу. Сейчас в `Segment.audio_duration` (`src/tts.py`) попадает длительность вместе с этим молчанием, и `merge.py` считает сегмент длиннее, чем он реально звучит — лишний раз триггерит `atempo`.

**Контекст.** Нейронные TTS-модели (включая Silero) обучены на парах «текст ↔ аудио», в которых записи студийные и обычно содержат 100-300 мс тишины до и после фразы. Эта тишина воспроизводится моделью как часть «звукового почерка». На длинном связном тексте это безвредно, но в нашем случае каждый сегмент — отдельный синтез, и хвостовое молчание считается «голосом», который надо втиснуть в окно. Если реально звучащая речь — 4.0 с, а WAV длится 4.4 с (0.4 с хвоста — пустой звук), `merge.py` видит overshoot и прогоняет `atempo`, ускоряя голос там, где можно было просто обрезать тишину.

**Идея.** После `silero.apply_tts(...)` обрезать хвостовое (и, опционально, ведущее) молчание перед сохранением WAV. Два варианта:
- `pydub.silence.detect_leading_silence` на инвертированном аудио — чисто Python, без новых зависимостей. `pydub` — обёртка над ffmpeg для работы с аудио в Python; функция возвращает мс ведущего молчания, через инверсию определяем хвостовое.
- `ffmpeg silenceremove=stop_periods=1:stop_silence=0.1:stop_threshold=-40dB` — фильтр ffmpeg, точнее обрабатывает шум, но добавляет ffmpeg-вызов на сегмент.
- _bluez-dubbing использует **Silero VAD** (`snakers4/silero-vad` через `torch.hub`) для той же задачи — `audio_processing.py:trim_audio_with_vad(several_seg=False)` обрезает только хвост._
- _Третий вариант реализации: VAD-классификатор «речь/тишина» точнее на шумных записях, чем энергетический threshold у `pydub`/`silenceremove`. Зависимость уже близкая — Silero TTS у нас и так грузится через `torch.hub`._

**Файлы.** `src/tts.py` — после `torchaudio.save` пройтись по WAV и переписать.

**Тонкость.** Не резать слишком агрессивно, иначе слово закроется в неловкий стык. Порог −40 дБ и окно 100 мс — безопасный старт (тише обычной речи, громче типичного фонового шума записи).

**DoD.** На боевом видео — суммарное `audio_duration` всех сегментов уменьшается на 5-15%, число `atempo`-стрейчей падает (мерить по `merge.py` диагностике).

#### 12. Persist Segment metadata в JSON + incremental rerun — _open-dubbing/utterance.py:105-125_

**Сложность: 35/100 · Эффект: 20/100**

**Проблема.** Если поправить один кривой перевод — приходится прогонять весь пайплайн заново (или хакать debugger'ом). TTS-синтез на 25-минутной лекции ≈ десятки секунд, перевод ≈ минуты.

**Контекст.** Сейчас наш пайплайн (`src/dub.py`) — линейный одноразовый прогон: транскрипция → перевод → синтез → merge, всё in-memory, ничего не сохраняется на диск кроме финального видео и WAV-файлов сегментов в /tmp. Это значит: чтобы поменять один перевод, нужно либо запускать всё заново (минуты), либо лезть в код. Open-dubbing хранит «состояние» пайплайна в `utterance_metadata.json` рядом с видео — это словарь со всеми сегментами и их полями. Каждое поле хэшируется, и при повторном запуске система видит «у этого сегмента изменился `translated` → нужно пере-синтезировать; у остальных не изменилось → берём готовый WAV из прошлого прогона». Превращает пайплайн из batch-утилиты в редактируемый артефакт.

**Идея.** После каждого этапа `dub.py` сериализовать `list[Segment]` в `<video>_segments.json` рядом с видео. При повторном запуске:
1. Если JSON есть — загрузить сегменты вместо транскрипции/перевода.
2. По флагу `--update` пере-синтезировать только те сегменты, у которых `translated` отличается от того, что было при последнем синтезе (хранить `translated_at_synthesis` или хэш).
3. Финальный `merge.py` всегда прогоняется заново (он быстрый).

Workflow: `dub.py video.mp4` → открыть JSON в редакторе → поправить кривой перевод в одной строке → `dub.py video.mp4 --update` → через секунды получить обновлённое видео.

**Файлы.** `src/segment.py` — `to_dict`/`from_dict`. `src/dub.py` — load/save и логика `--update`. Возможно отдельный `src/persistence.py`.

**Тонкость.** Не делать SHA256 каждого поля, как у них — overkill для нашего скоупа. Достаточно хранить `translated_at_synthesis: str` рядом с `audio_path` и сравнивать строкой.

**DoD.** Сценарий: запуск, правка одной строки в JSON, `--update` — пере-синтезировался ровно один сегмент, остальные взяты из кеша. Финальное видео отличается от предыдущего только в этом месте.

### B. Новые фичи

#### 13. Demucs vocal separation + 3-track mix — _open-dubbing/demucs.py_

**Сложность: 50/100 · Эффект: 75/100**

**Проблема.** Сейчас `merge.py` делает `amix` русского TTS поверх оригинальной дорожки — английский голос всегда подмешан под русским. Для лекции терпимо, для фильма/влога — фатально.

**Контекст.** Demucs (Deep Extractor for Music Sources) — нейросеть от Meta AI Research для задачи **source separation**: разложить аудио-микс на «дорожки» (вокал, бас, барабаны и т.д.) так, как если бы у тебя были исходные мульти-треки записи. Архитектура `htdemucs` (Hybrid Transformer Demucs, текущий SOTA) работает одновременно во временной и спектральной области. Режим `--two-stems=vocals` склеивает всё кроме голоса в один трек: на выходе `vocals.wav` (только речь/пение) и `no_vocals.wav` (всё остальное — музыка, фоновые звуки, шум зала, реверберация). Для дубляжа это ровно то, что нужно: убираем оригинальный голос, оставляем атмосферу, поверх кладём свой. На студийных записях с чистой речью разделение почти идеальное; на «грязных» (улица, ресторан) — могут быть артефакты.

**Идея.** Новый шаг между извлечением аудио и сборкой:
1. `demucs --two-stems=vocals` на исходном аудио → `vocals.wav` + `no_vocals.wav` (фон: музыка, SFX, шум).
2. `merge.py` микширует: `[no_vocals.wav] + [russian_segments]`. Английский вокал полностью удалён, фон сохраняется.
3. Опционально: ducking — автоматическое приглушение фона на 6-9 дБ во время русской речи через ffmpeg `sidechaincompress` (фон = «вход» компрессора, русский голос = «триггер»). Так голос не «тонет» в музыке.

**Файлы.** Новый `src/separate.py` (тонкая обёртка над `demucs` CLI или Python API). Изменить `src/merge.py` — принимать background track. `config.yaml` — флаг `separation.enabled`.

**Тонкость.** Demucs htdemucs ~80 MB модель, на CPU ≈1-2× длительности видео (т.е. 25-минутное видео обрабатывается 25-50 минут), на GPU — ~0.1×. Сделать опциональным шагом через `config.yaml` — на студийных лекциях разделение скорее всего не нужно, на видео с музыкой/фоном — обязательно.

- _bluez-dubbing использует тот же стек через пакет `audio-separator` (см. #19) — выбор UVR или Demucs остаётся за нами._
- _При включённой #15 diarization фон логично разделять и по фразам тоже не нужен — главное сохранить стерео-картину между сегментами._

**DoD.** На видео с фоновой музыкой — финальный дубляж содержит русский голос + оригинальную музыку, без следов английской речи. Метрика: записать spectrogram до/после, английские форманты в финале отсутствуют.

#### 14. SRT subtitle generation — _open-dubbing/subtitles.py_

**Сложность: 10/100 · Эффект: 25/100**

**Контекст.** SRT (SubRip Subtitle) — самый распространённый формат субтитров: plain-text файл с блоками `номер / таймкод-старт --> таймкод-конец / текст / пустая строка`. Видеоплееры (VLC, mpv, QuickTime, YouTube) автоматически подхватывают `<videoname>.srt` рядом с видео или вшитую дорожку с тегом языка. У нас уже есть всё нужное — `Segment.start/end/original/translated` — генерация это форматирование строк, никакой ML.

**Идея.** После транскрипции и перевода сохранять два файла рядом с видео:
- `<video>.en.srt` — оригинал
- `<video>.ru.srt` — перевод

Опционально — вшить в выходное видео через ffmpeg:
- **soft subs** (`-c:s mov_text` для mp4) — отдельная дорожка субтитров внутри контейнера, плеер может включить/выключить.
- **hard subs** (burn-in через фильтр `subtitles=...`) — субтитры впечатаны в видеоряд, неотключаемы, но работают везде.

**Файлы.** Новый `src/subtitles.py` (~30 строк, простой SRT writer). Вызов из `src/dub.py` после translate.

**Тонкость.** SRT требует таймкоды формата `HH:MM:SS,mmm` — запятая вместо точки в десятичной дроби, частая ошибка. Длинные сегменты (>5 сек) для читаемости стоит разбивать, но это already-solved нашим resegmenter'ом — он и так режет по фразам.

**DoD.** Открыть финальное видео в плеере, переключить дорожку субтитров → видны оба языка. SRT-файлы валидны (проверить через `ffprobe -i video.mp4` — должны отображаться субтитровые streams).

#### 15. PyAnnote speaker diarization — _open-dubbing/speech_to_text.py_

**Сложность: 50/100 · Эффект: 60/100**

**Проблема.** Сейчас весь дубляж озвучивается одним голосом Silero (`speaker` из config). На диалогах/интервью это звучит как один человек, читающий пьесу за всех — теряется ощущение разных людей.

**Контекст.** Диаризация (speaker diarization) — задача «кто говорит когда»: разметить аудио на интервалы вида `(start, end, speaker_id)`, где `speaker_id` — анонимный кластер (`SPEAKER_00`, `SPEAKER_01`, …). Это **не** распознавание конкретных людей по голосу — система просто понимает «вот эти 3 минуты говорил один человек, а вот эти 30 секунд — другой». PyAnnote — open-source библиотека от французского INRIA, де-факто стандарт для diarization. Модель `pyannote/speaker-diarization-3.1` использует двухступенчатый pipeline: сегментация голосовой активности → embedding каждого сегмента в вектор → кластеризация векторов. Это «gated model» на HuggingFace — нужно один раз согласиться с условиями использования на странице модели и получить access token (бесплатно).

**Идея.** Новый шаг между транскрипцией и переводом:
1. Прогнать оригинальное аудио через `pyannote.audio` `Pipeline.from_pretrained("pyannote/speaker-diarization-3.1")` → список `(start, end, speaker_id)`.
2. Каждому `Segment` присвоить `speaker_id` по максимальному пересечению с диаризацией (`Segment` от Whisper не знает кто говорит, тут мы это знание добавляем).
3. На этапе TTS — маппинг `speaker_id → silero_speaker`, разные люди → разные голоса.

**Файлы.** Новый `src/diarize.py`. Расширить `Segment` полем `speaker_id: str | None` (`src/segment.py`). `src/tts.py` — выбирать голос по `speaker_id`. `config.yaml` — флаг `diarization.enabled` + HF token из env.

**Тонкость.**
- **HF token.** Pyannote требует `HUGGINGFACE_TOKEN` в env + ручное согласие на странице модели. Документировать в README.
- **Лимит голосов Silero.** В `v4_ru` — 5 голосов (`aidar`, `eugene`, `baya`, `kseniya`, `xenia`). Если в видео >5 спикеров — нужна стратегия (round-robin / самые громкие 5 + остальные одним).
- **Качество на overlap.** На пересекающейся речи (двое говорят одновременно) PyAnnote выдаёт сегменты обоих, но их объединение в наш Segment-модель — отдельный вопрос. Стартовать с предположения «overlap игнорируем, берём dominant speaker».
- **Скорость.** На CPU ≈0.3-0.5× длительности видео, на GPU быстрее.
- _bluez-dubbing встраивает diarization прямо в WhisperX runner_1 (`whisperx.diarize.DiarizationPipeline.from_pretrained(...)`) — то есть запускает её как часть ASR-пайплайна, а не отдельным шагом. Если переходим на WhisperX (см. #24), естественно совместить._
- _В обмен на это получаем word-level alignment + diarization за один проход — экономия инференс-времени и общая модель для align/diarize._

**DoD.** На диалоговом видео (2-3 спикера) — каждому в финальной дорожке достаётся свой Silero-голос. При смене спикера в кадре — переключение голоса в дубляже.

#### 16. Voice gender classifier для умного выбора голоса — _open-dubbing/voice_gender_classifier.py_

**Сложность: 35/100 · Эффект: 30/100**

**Проблема.** Допустим #15 разметил спикеров. Дальше нужно решить: какой Silero-голос дать `SPEAKER_00`, какой — `SPEAKER_01`. Простейший round-robin может присвоить мужчине женский голос — это режет ухо сильнее, чем «просто один голос на всех».

**Контекст.** Wav2Vec2 — нейросеть от Meta, обученная self-supervised на огромных корпусах речи без разметки. Базовая модель учится восстанавливать замаскированные кусочки аудио, в процессе формируя богатые представления (embeddings) звука речи. На эти embeddings сверху можно «накатить» небольшую классификационную голову и fine-tune на любой задаче со скромным датасетом. Модель `audeering/wav2vec2-large-robust-24-ft-age-gender` — именно такой fine-tune, предсказывает возраст (число) и пол (probability M/F) по короткому фрагменту голоса. Точность по полу на чистой речи ≈95%+, не идеально на детских голосах и крайних случаях.

**Идея.** После #15 для каждого `speaker_id`:
1. Найти самый длинный непрерывный фрагмент его речи в оригинальном аудио (≈3-10 сек достаточно для классификатора).
2. Прогнать через wav2vec2-age-gender → `Male` / `Female`.
3. Маппинг по полу: M → один из `aidar`/`eugene`, F → один из `baya`/`kseniya`/`xenia`. Внутри пола — round-robin или жребий по кластеру.

**Файлы.** Новый `src/voice_classifier.py`. Расширение логики маппинга в `src/tts.py` (вместо прямого `speaker_id → voice` — двухуровневое `speaker_id → gender → voice`).

**Тонкость.**
- **Размер модели.** wav2vec2-large ≈1.2 GB. Lazy load только если включена `diarization` И `gender_classifier`.
- **Бинарность.** Классификатор даёт строго M/F, не учитывает индивидуальный тембр. Двое мужчин получат разные мужские голоса (по round-robin), но не «похожие на оригинал» — для этого нужен voice cloning (см. #10 XTTS/F5-TTS).
- **Альтернатива без классификатора.** Greedy round-robin по всем 5 голосам Silero без оглядки на пол — проще, но даст mismatch. Имеет смысл как первая итерация multi-speaker, gender classifier — следующая.

**DoD.** На видео с мужским и женским спикером — мужчине в дубляже достался мужской голос, женщине — женский. На сравнении: запуск без gender classifier (round-robin) на том же видео — может присвоить наоборот.

#### 17. Явный carry-over silence + cap в merge.py — _ViDubb/inference.py:508–545_

**Сложность: 20/100 · Эффект: 40/100**

**Контекст.** Сейчас наш `merge.py` использует «эффективное окно» `next.start − this.start` для расчёта `atempo` — естественные паузы между сегментами уже учтены. Но логика монолитная: окно либо влезло, либо триггерим стрейч до `ATEMPO_MAX=1.5`. ViDubb делает то же самое явнее и с дополнительным контролем: считает `theta = lo / lt` (где `lo` — оригинальная длительность, `lt` — длительность TTS), отдельно отслеживает `previous_silence_time` (сколько паузы из gap'а между сегментами «передаётся» следующему как фора), и кэпит этот перенос: `if natural_silence >= 0.8: previous_silence_time = 0.8` — не более 0.8 с. Это значит: даже если gap большой (например 3 с между предложениями), для расчёта стрейча следующего сегмента берётся максимум 0.8 с — остаток остаётся естественной паузой и не «съедается» под ускорение.

**Идея.** Переписать `merge.py` с явным трекингом `prev_carry_silence` (по аналогии с ViDubb), вынести cap в `config.yaml` (`merge.max_carry_silence: 0.8`). Это даст две вещи: (а) предсказуемое поведение на длинных межсегментных паузах — мы не будем растягивать предыдущий TTS до конца паузы, оставим дыхание; (б) более прозрачный диагностический вывод (сколько секунд паузы реально использовано под стрейч, сколько осталось).

**Связь с #5.** Этот пункт **не заменяет** #5 cascading playback. #5 решает обратную проблему: TTS закончился раньше окна → сместить старт следующего сегмента раньше его номинального `start`. #17 решает: окна не хватает → взять до 0.8 с из последующей паузы. Оба механизма комплементарны и могут жить вместе (флаги в config).

**Файлы.** `src/merge.py`, `config.yaml`, тесты.

**DoD.** На видео с длинной паузой между сегментами (>2 с) и небольшим overshoot предыдущего TTS — пауза в финальной дорожке остаётся ≥1.2 с (не «проедается» полностью под стрейч). Метрика: средний `previous_silence_time` в логах ≤ 0.8.

#### 18. Defensive truncate-to-N-words перед TTS — _ViDubb/inference.py:497, truncate_text_

> **Status (2026-05-03):** Отклонено аналитически, без эмпирического прогона. В текущей архитектуре триггер физически не достижим: `transcribe._resegment` имеет hard cap `max_duration=8.0` (≤ 8 сек), `group.py` — `max_duration=12.0` (≤ 12 сек). 12 сек × ~3 слова/сек английской речи = ~36 EN слов, × 1.5 на русскую expansion = ~54 RU слов. Чтобы перевалить порог 50–80 слов, нужно либо чтобы LLM при `temperature=0` утроил длину (крайне маловероятно), либо чтобы upstream-кэпы увеличили в 2–3 раза. **Эффект пересмотрен с 10/100 до ~0/100** — не реализовывать в текущем виде. **Реактивировать**, если: (а) `transcribe.max_duration` или `group.max_duration` поднимут до ≥25 сек; (б) переход на voice-cloning TTS (#10, XTTS) — он чувствителен к длинному входу (мусор/таймаут), guard становится оправдан; (в) появятся реальные кейсы overshoot LLM ×2-3 на боевом видео.

**Сложность: 5/100 · Эффект: ~~10~~ → ~0/100** (после анализа)

**Контекст.** ViDubb перед `tts_to_file` обрезает текст до 50 слов через `truncate_text(...)` — защита от того, что LLM или ASR вернут аномально длинный текст для одного сегмента (например, ASR не нашёл паузу и склеил два предложения в одну запись). Без этой защиты XTTS на длинном куске либо генерит мусор, либо падает по таймауту. У нас сейчас защиты нет: если `seg.translated` оказался в 200 слов, Silero пойдёт синтезить 200 слов, и `merge.py` потом будет драматически стрейчить.

**Идея.** В `src/tts.py` перед `silero.apply_tts` проверять длину `seg.translated` в словах. При превышении порога (например 60–80 слов на сегмент при нашем `max_duration=12s`) — логировать warning и обрезать до ближайшей границы предложения. Это сигнал, что upstream (ресегментация или перевод) дал кривой результат — лучше потерять хвост, чем ломать всю дорожку.

**Файлы.** `src/tts.py`, опционально вынести порог в `config.yaml` (`tts.max_words_per_segment: 60`).

**Тонкость.** Это защита от багов, не штатный путь. Если триггерится регулярно — чинить не TTS, а ресегментацию (`src/transcribe.py`). Логировать с ясным префиксом (`[tts] truncated segment X from Y to Z words`), чтобы было видно в выводе.

**DoD.** Подложный сегмент с переводом в 100 слов → лог-предупреждение, синтез только первой части, пайплайн не падает.

#### 19. Audio-Separator (UVR-MDX) как альтернатива Demucs в #13 — _ViDubb/inference.py:600–614_

**Сложность: 25/100 · Эффект: 75/100**

ViDubb решает ту же задачу через пакет [`audio-separator`](https://github.com/nomadkaraoke/python-audio-separator) с моделью `2_HP-UVR.pth` (семейство UVR-MDX-Net, разработано community Ultimate Vocal Remover). **Файлы.** Те же, что у #13 (`src/separate.py`).

#### 20. Emotion preservation через SpeechBrain IEMOCAP — _ViDubb/inference.py:364–370, 468–470_

**Сложность: 50/100 · Эффект: 50/100**

**Проблема.** Дубляж монотонный: даже если оригинал произнесён эмоционально (раздражённо, радостно, печально), русский TTS выдаёт нейтральный тон. На лекциях терпимо, на интервью/драматическом контенте — плоско.

**Контекст.** SpeechBrain — open-source toolkit для речевых задач от исследователей INRIA/Mila. Модель `speechbrain/emotion-recognition-wav2vec2-IEMOCAP` — fine-tune Wav2Vec2 на корпусе IEMOCAP (Interactive Emotional Dyadic Motion Capture, ~12 часов диалогов с эмоциональной разметкой от USC). Классифицирует фрагмент аудио в одну из 4 категорий: `Neutral`, `Angry`, `Happy`, `Sad`. Точность на студийной речи ≈70–75%, что для нашего сценария «прокинуть подсказку в TTS» достаточно — даже если 25% сегментов получают неточную метку, общая интонационная картина становится разнообразнее монотонного `Neutral`.

ViDubb прогоняет каждый сегмент оригинального аудио через этот классификатор и передаёт результат в XTTS v2 как параметр `emotion=...`. XTTS обучен учитывать эту метку и менять интонацию синтеза.

**Идея.** Добавить шаг между транскрипцией и TTS:
1. Для каждого `Segment` извлечь окно оригинального аудио по `start`/`end`, прогнать через SpeechBrain IEMOCAP.
2. Положить результат в новое поле `Segment.emotion: str | None`.
3. Передавать в TTS-движок, который поддерживает emotion-условие.

**Связь с TTS.** Silero v4_ru **не принимает** emotion-параметр — для него поле бесполезно. Имеет смысл **только** в связке с XTTS v2 / F5-TTS / Fish Speech (см. секцию TTS-экспериментов). Поэтому пункт логически зависит от перехода на voice cloning движок.

**Файлы.** Новый `src/emotion.py`, расширить `Segment` полем `emotion`, интегрировать в `src/tts.py` (если выбран совместимый движок).

**Тонкость.** SpeechBrain тянет дополнительные зависимости (`torch`, `speechbrain`, чекпойнт ≈350 MB). Lazy-load, грузить только при `emotion.enabled: true` в config. На очень коротких сегментах (<1 с) классификатор шумит — для таких просто оставлять `Neutral`.

**DoD.** На видео с явно эмоциональными фразами (восклицания, грустные паузы) — `Segment.emotion` отражает оригинал, TTS-выход звучит соответственно (субъективная проверка). Метрика: распределение классов на 100 сегментах не вырождено в `Neutral=100%`.

#### 21. Wav2Lip lip-sync — _ViDubb/inference.py:638–642 + Wav2Lip/_

**Сложность: 85/100 · Эффект: 70/100**

**Проблема.** Дубляж рассинхрон с губами в кадре. Терпимо для лекций с фронтальной камерой издалека, неприятно для крупных планов / интервью / влогов.

**Контекст.** Wav2Lip — модель из статьи «A Lip Sync Expert Is All You Need for Speech to Lip Generation In The Wild» (Prajwal et al., ACMMM 2020, IIIT Hyderabad). Двухголовая GAN-архитектура: генератор перерисовывает нижнюю половину лица под целевое аудио (mel-спектрограмму), дискриминатор-эксперт по lip-sync (предобученный SyncNet) штрафует за рассинхрон. Работает кадр-в-кадр: на вход — оригинальный кадр + 0.2 сек целевого аудио, на выход — кадр с пересобранным ртом. Вес `wav2lip_gan.pth` ≈400 MB, инференс на GPU ≈0.1× длительности видео, на CPU практически непригодно. Модель 2020 года, есть артефакты на крупных планах и нестандартных ракурсах, но всё ещё де-факто стандарт open-source lip-sync. Альтернативы — SadTalker (talking head из одного фото), MuseTalk (2024, лучше качество), но интеграционно сложнее.

ViDubb использует двухступенчатую детекцию лица: Haar Cascade (OpenCV) для черновой обрезки → DeepFace ArcFace для извлечения embeddings → cosine-similarity кластеризация для нахождения «одного и того же лица» спикера на разных кадрах. Это позволяет в multi-speaker видео правильно подавать Wav2Lip только на лицо текущего говорящего.

**Идея.** Опциональный финальный шаг после `merge.py`: если `lipsync.enabled: true` — прогнать собранное видео через Wav2Lip с уже синтезированной русской дорожкой. Шаг должен быть отдельным CLI-флагом и явно опциональным — это самая тяжёлая часть пайплайна.

**Файлы.** Новый `src/lipsync.py` (обёртка над Wav2Lip CLI или Python API), интеграция в `src/dub.py` под флагом. Возможно отдельный submodule в репо или зависимость через git-clone.

**Тонкость.**
- **GPU обязателен.** На CPU 25-минутное видео обрабатывается часами. Без GPU пункт не имеет смысла.
- **Лицензия.** Wav2Lip — research-only license от IIIT (не для коммерческого использования). Для open-source/личных экспериментов ок, для продукта — блокер.
- **Качество на фоне современных моделей.** Артефакты на быстром движении/повороте головы. Если приоритет — качество, смотреть в сторону MuseTalk или коммерческих API (HeyGen, Synthesia). Wav2Lip — это «работает прямо сейчас и бесплатно», не «выглядит идеально».
- **Multi-speaker нужен diarization (#15).** Без диаризации Wav2Lip будет переписывать губы любому лицу в кадре, включая немых слушателей в аудитории.

**Связь с другими пунктами.** Хорошо работает поверх стека #15 (diarization) + #16 (gender) + voice cloning из TTS-секции. Без них lip-sync на multi-speaker видео даст рассинхрон или применится не к тому лицу.

**DoD.** На видео с крупным планом одного спикера — губы в финальной дорожке движутся в такт русскому TTS (визуальная проверка, можно SyncNet-метрикой). Время инференса на GPU укладывается в разумный budget.

#### 22. Rubberband вместо ffmpeg atempo для time-stretch — _bluez-dubbing/audio_processing.py concatenate_audio_

**Сложность: 20/100 · Эффект: 40/100**

**Контекст.** Сейчас `merge.py` использует ffmpeg-фильтр `atempo` (с потолком 1.5×) для ускорения TTS, когда он не влез в окно. `atempo` сохраняет питч, но на больших коэффициентах (>1.25×) даёт характерную «булькающую» артефакцию — слышно, что речь ускорена. Это и есть причина, почему мы держим cap. **Rubberband** — отдельная библиотека для high-quality time-stretching и pitch-shifting, которой пользуется bluez-dubbing через Python-обёртку `pyrubberband`. Алгоритм фазовый, на коротких сегментах речи звучит существенно чище ffmpeg на тех же ratio (можно безопасно идти до 1.4–1.5× без слышимых артефактов) и **поддерживает обе стороны**: не только ускорение, но и замедление (ratio < 1) — то есть растягивать слишком короткий TTS, чтобы заполнить окно, а не оставлять провал.

**Идея.** Заменить в `merge.py` фильтр `atempo` на pre-pass через `pyrubberband.time_stretch(audio, sr, ratio)` для каждого сегмента. После Rubberband сегмент сохраняется в WAV нужной длительности и подаётся в ffmpeg уже в готовом виде (`amix`/`adelay` без time-stretch фильтров). Это меняет архитектуру `merge.py`: time-stretch перестаёт быть частью ffmpeg-графа и становится отдельным шагом на этапе TTS-постпроцессинга.

**Плюсы.**
- Лучше качество звука на ratio >1.25 — снимает причину держать `ATEMPO_MAX=1.5` как защитный потолок.
- Двусторонний stretch: можно растягивать короткие TTS, заполняя «дырки», вместо оставления тишины. Полезно для длинных английских фраз с короткими русскими аналогами (антоним основной проблемы — недостаточно частое явление, но встречается).
- Чище разделение responsibilities: time-stretch на уровне аудио, ffmpeg только склейка.

**Минусы.**
- Новая зависимость: системный `rubberband` (`brew install rubberband`) + `pip install pyrubberband`.
- Дополнительный I/O: каждый сегмент пере-сохраняется после стрейча.
- Усложняет связку с #17 (carry-over silence cap) — расчёт `theta` нужно делать до Rubberband, не в фильтр-графе.

**Файлы.** `src/merge.py` — переписать stretch-логику. `config.yaml` — флаг `merge.stretch_engine: atempo|rubberband`, чтобы можно было откатиться.

**Связь.** Дополняет #5 и #17 — все три про обработку рассинхрона длительности, но на разных уровнях. Rubberband улучшает **качество** стрейча, #5/#17 — **стратегию** (когда стрейчить, что переносить в паузу).

**DoD.** A/B на сегменте с ratio 1.4: rubberband-версия звучит чище, нет «булькания» atempo. Замер: можно поднять `ATEMPO_MAX_EQUIV` до 1.5–1.6 без слышимых артефактов на ушах.

#### 23. Weighted silence distribution между сегментами — _bluez-dubbing/audio_processing.py concatenate_audio_

**Сложность: 25/100 · Эффект: 30/100**

**Контекст.** В bluez-dubbing после генерации всех TTS-сегментов перед склейкой считается общая длительность исходного видео и общая длительность готовых TTS. Разница (свободная пауза) распределяется между сегментами **пропорционально количеству non-punctuation токенов** — то есть длинная фраза получает больше «дыхания» перед собой, чем короткий вкрапленный «yeah». У нас сейчас межсегментные паузы фиксированы — те, что были в оригинале (`next.start − this.end`), без учёта реальной длины русского TTS. Если дубляж в сумме короче оригинала — лишняя тишина равномерно распределяется через `adelay`, а если длиннее — мы сжимаем. И в том, и в другом случае паузы между фразами могут оказаться неестественно равномерными.

**Идея.** В `merge.py` после расчёта всех TTS-длительностей: посчитать `slack = video_duration − sum(audio_durations)`. Если `slack > 0`, распределить его между сегментами с весом `weight_i = word_count_i`, то есть сегмент с 12 словами получит в 4 раза больше дополнительной паузы перед собой, чем сегмент с 3 словами. Это эмулирует естественное «дыхание» — перед длинной мыслью человек делает паузу подлиннее.

**Связь с #5/#17.** Эта механика работает в режиме «перевод суммарно короче оригинала»; #5/#17 — в режиме «дольше». Не конфликтуют, но реализуются вместе как единая «timing strategy» в `merge.py`.

**Файлы.** `src/merge.py`, `config.yaml` (флаг `merge.weighted_silence: true`).

**Тонкость.** Очевидный риск — рассинхрон с видео, если к концу длинного видео накапливается много «лишней» паузы. Логично применять только в окне (например, скользящее окно из 20 сегментов с локальным `slack`), а не глобально.

**DoD.** На видео, где русский дубляж суммарно на 5 с короче оригинала — паузы перед длинными фразами заметно длиннее, чем перед короткими. Метрика: дисперсия inter-segment silence ↑.

#### 24. Word-level timestamps через WhisperX (или native faster-whisper) — _bluez-dubbing/services/asr/whisperxModel_

**Сложность: 30/100 · Эффект: 25/100**

**Контекст.** Сейчас `transcribe.py` берёт у faster-whisper только сегменты — `start/end/text`. У нас уже включён `word_timestamps=True`, но мы используем word-таймстемпы только для resegment-логики и затем выкидываем. WhisperX от bluez-dubbing идёт дальше: после Whisper-транскрипции прогоняет **forced alignment** через wav2vec2 (модель `WAV2VEC2_ASR_LARGE_LV60K_960H` для английского), получая word-level timings c sub-Whisper точностью (Whisper сам по себе на word-уровне шумит, особенно на быстрой речи). Результат — `Segment.words: list[Word(start, end, text, score)]` с очень точными границами слов.

**Что это нам даёт.** Само по себе — **немного**: для нашего текущего пайплайна (сегмент = единица перевода и синтеза) точные word-таймстемпы не нужны, мы оперируем уровнем фраз. Но это enabler для нескольких будущих фич:
- **Точная пересегментация** перевода: SophisticatedAligner (см. #25) умеет резать длинный сегмент на под-сегменты по фразовым границам, опираясь на word-timings.
- **Lip-sync** (#21): Wav2Lip нуждается в точных границах фонем/слов для правильной анимации губ — без этого губы в дубляже либо опаздывают, либо обгоняют звук.
- **Karaoke-style subtitles**: word-by-word подсветка, если когда-нибудь будем экспортировать ASS-субтитры.

**Идея.** Два варианта реализации, по убыванию сложности:
1. **Native faster-whisper**: уже в `word_timestamps=True`. Сохранять `segment.words` в `Segment` как новое поле `words: list[Word] | None`. Никаких новых зависимостей. Точность хуже WhisperX, но на наших задачах достаточно для начала.
2. **WhisperX**: установить `whisperx`, заменить `WhisperModel` на `whisperx.load_model() + whisperx.align()`. Требует wav2vec2-чекпойнт (~1.2 GB для английского), даёт более точные timings и встроенный pyannote runner для diarization (см. #15) — два хода одной картой.

**Файлы.** `src/segment.py` — добавить поле `words`. `src/transcribe.py` — сохранять word-timings из faster-whisper или переключиться на whisperx. `config.yaml` — `transcription.engine: faster-whisper|whisperx`.

**Тонкость.** На CPU WhisperX заметно медленнее faster-whisper (запускает второй проход alignment), и большинство сегментов в нашем сценарии всё равно будут переведены и синтезированы как целое. Имеет смысл переходить **в связке с** #15 (diarization) и/или #25 (realignment), а не сам по себе.

**DoD.** `Segment.words` заполнено, в `tests/` есть тест на корректность форматов. Реальная польза — после интеграции с одной из upstream-фич.

#### 25. SophisticatedAligner / ProportionalAligner для пересегментации перевода — _bluez-dubbing/libs/common-schemas/utils.py_

**Сложность: 70/100 · Эффект: 35/100**

**Контекст.** В bluez-dubbing есть нетривиальный модуль (~1400 строк) для **realignment перевода**: после того как вся английская дорожка переведена в русскую, систему интересует, как разбить русский текст на сегменты с теми же временными границами, что у оригинала. У них два алгоритма:

- **ProportionalAligner** — простой, для родственных языков (en→fr, en→de): распределяет русские токены пропорционально длительности оригинальных сегментов.
- **SophisticatedAligner** — сложный, для языков с радикально другим порядком слов (en→ru, en→zh, en→ja): использует **SimAlign** — нейросеть на BERT-эмбеддингах, которая считает alignment между токенами на разных языках (какое русское слово соответствует какому английскому). Затем переразбивает русский текст так, чтобы каждый сегмент переводил соответствующий ему оригинальный, плюс **«determiners awareness»**: на границе сегментов не оставляет артикли/предлоги отдельно от их существительного (например, не разрезает «in the morning» так, чтобы «the» оказался в одном сегменте, а «morning» — в следующем).

**Что это нам даёт.** Прямо сейчас мы **не делаем** пересегментацию: один английский сегмент → один русский сегмент → один TTS-WAV. Если LLM выдала перевод, который семантически охватывает чуть больше или меньше, чем оригинал — нам всё равно, мы синтезируем как есть. Realignment имел бы смысл, **если** мы стали переводить всю дорожку целиком (а не посегментно), и потом разрезать результат — это даёт более естественный русский перевод (LLM не зажата в рамки одного сегмента) ценой сложности.

**Идея.** Долгосрочный пивот: переход с per-segment перевода на whole-text перевод + realignment. Маленькими шагами:
1. Сохранить текущий per-segment перевод как baseline.
2. Реализовать ProportionalAligner — просто, на токенах.
3. Если на боевых видео получаем более естественный перевод — заходить в SimAlign / SophisticatedAligner.

**Файлы.** Новый `src/realign.py`. Изменения в `src/translate.py` (whole-text режим) и `src/dub.py` (порядок этапов).

**Связь.** Требует #24 (word-timestamps) для алгоритма. Дополнительно, на CJK-языках понадобятся специфичные tokenizers (MeCab/jieba) — у нас целевой язык русский, эта часть не нужна.

**Тонкость.** Большая инженерная работа с сомнительной отдачей для нашего сценария (en→ru, не CJK). Скорее «теоретически интересно», чем «возьму на следующей неделе».

**DoD.** A/B: «whole-text + realign» vs «per-segment» — субъективно более естественные русские формулировки на 5+ роликах разной тематики, без потери синхронизации.

### B. Новые фичи

#### 26. Document-level context pre-pass: тематика + сущности + glossary — _наша уникальная фича_

**Сложность: 25/100 · Эффект: 70/100**

**Проблема.** У `translate.py` сейчас два слепых пятна на уровне всего видео:

1. **Нет понимания, про что вообще видео.** Sliding window N=3 даёт только последние 3 сегмента — на 10-й минуте лекции LLM не знает, что речь про Kubernetes / античную философию / gamedev. Отсюда промахи на многозначных словах (`pod`, `attention`, `model`, `bias`, `scale`), неправильный регистр (академический vs разговорный) и непоследовательные местоимения для введённых ранее сущностей.
2. **Нет явных инструкций по терминологии.** Gemma переводит «Kubernetes» как «Кубернетс» (норм), «GPT-4» как «ГПТ-4» (плохо), «Claude» — то «Клод», то «Claude» (непоследовательно), а «attention» в ML-контексте превращает во «внимание» вместо оставления как есть. Особенно болезненно на технических лекциях.

Обе проблемы решаются одним и тем же механизмом — общим контекстом, который кладётся в `SYSTEM_PROMPT` _до_ старта поштучного перевода.

**Контекст.** Ни в одном из изученных проектов (open-dubbing, ViDubb, bluez-dubbing, AutoDub) **нет** ни глоссария, ни document-level context. Все полагаются на качество MT (DeepL/Google) и принимают что есть. У нас преимущество — **локальная LLM с управляемым system prompt'ом**, в который можно положить и описание видео, и явные инструкции «не переводить».

**Идея.** Один extraction-проход после транскрипции, который возвращает структурированный JSON и подаётся в `SYSTEM_PROMPT` каждого сегмента:

1. **Один вызов Ollama** на полный (или sample-обрезанный до ~3000 токенов) транскрипт с промптом «Проанализируй транскрипт. Верни JSON со схемой:»
   ```json
   {
     "topic": "Kubernetes networking deep dive",
     "domain": "cloud infrastructure / DevOps",
     "register": "technical lecture",
     "key_entities": ["Cilium", "eBPF", "Envoy"],
     "glossary_keep_original": ["Kubernetes", "GPT-4", "eBPF", "Pod", "Service Mesh"],
     "glossary_transliterate": ["Hannah → Ханна", "Cilium → Цилиум"]
   }
   ```
2. **Инъекция в system prompt** перевода. К текущему промпту добавляются 4 строки: `Topic: {topic}. Domain: {domain}. Register: {register}. Do not translate: [...]. Transliterate: [...]`. Length hint и history N=3 остаются как есть.
3. **Опциональный manual override.** Рядом с видео можно положить `<video>.context.json` с любыми полями выше — они перезапишут авто-извлечённые. Если файла нет — извлекать полностью автоматически.

**Файлы.** Новый `src/context.py` (extract + load + merge с override). `src/translate.py` — расширить `SYSTEM_PROMPT` форматтером, принять `VideoContext` как опциональный параметр. `src/dub.py` — добавить шаг между `transcribe` и `translate`. `src/segment.py` — без изменений (контекст глобальный, не per-segment).

**Тонкость.**
- **Sample vs full transcript.** Для 25-минутного видео полный транскрипт — ~5K токенов, gemma4:e4b прожуёт без проблем. Для длинных подкастов (>1ч) — отдавать первые 2K + последние 1K токенов, этого достаточно для определения topic/domain.
- **Транслитерация vs сохранение.** Имена людей — транслитерация (Hannah → Ханна), бренды/код/аббревиатуры — сохранение латиницей (GPT-4, Kubernetes). Extraction-промпт сразу разделяет на два списка.
- **Бюджет на токены в per-segment вызовах.** Глобальный контекст добавляет ~150–250 токенов в каждый вызов перевода. На 200 сегментах это +40K токенов overhead, но Ollama локальная — стоимость только в скорости (~10–20% медленнее).
- **Риск систематического перекоса.** Если extraction ошибся в topic/domain, ВЕСЬ перевод поедет в неправильную сторону. Митигация: ограничить free-form поля до коротких фраз (JSON schema с `maxLength`), для glossary — отдельная валидация (строки только из латиницы / без знаков препинания).
- **Few-shot synergy** с #3: примеры в few-shot можно подбирать **по domain** из извлечённого контекста (для технического — технические примеры, для нарративного — нарративные).

**DoD.**
1. **Glossary exact-match**: на видео с >5 техническими терминами — до фикса 50%+ переведены/искажены, после — 90%+ сохранены в правильной форме (ground-truth список вручную).
2. **Document context**: на видео с многозначной лексикой (например, ML-лекция с `attention`, `model`, `head`) — слепое A/B сравнение «с контекстом vs без» из 5 случайных сегментов: версия с контекстом выигрывает в ≥4/5.

#### 27. Plug-in архитектура переводчика (Ollama / DeepL / Claude API) — _bluez-dubbing translation/models registry_

**Сложность: 25/100 · Эффект: 60/100**

**Контекст.** Сейчас `translate.py` жёстко привязан к Ollama: `requests.post(ollama_url + "/api/chat", ...)`. bluez-dubbing использует **registry pattern** — каждая модель перевода (deep_translator, m2m100, …) реализует общий интерфейс и регистрируется в YAML-конфиге. У них таким образом одно изменение config'а переключает Google → DeepL → ChatGPT → m2m100. Мы можем сделать тот же паттерн.

**Идея.** Абстракция `TranslationProvider` с одним методом `translate(text: str, history: list, length_hint: int) -> str`. Реализации:
1. **OllamaProvider** — текущий код, переезжает за интерфейс.
2. **DeepLProvider** — через `deep_translator` (минимальная зависимость, поддержка 8+ облачных API). Качество en→ru объективно выше Ollama-моделей.
3. **ClaudeProvider** — через Anthropic API. Лучшее качество и контекстуальность, но платное и не локальное.
4. **AyaProvider** — Ollama + `aya-expanse:8b` (де-факто это вариант OllamaProvider с другой моделью; см. #28). Можно не выделять отдельно.

Конфиг:
```yaml
translation:
  provider: ollama  # ollama | deepl | claude
  ollama:
    model: gemma4:e4b
    url: http://localhost:11434
  deepl:
    api_key: ${DEEPL_API_KEY}
  claude:
    model: claude-sonnet-4-6
    api_key: ${ANTHROPIC_API_KEY}
```

**Плюсы.**
- Лёгкое A/B сравнение разных переводчиков на одном видео.
- Опциональный «премиум» режим (DeepL/Claude) для пользователей, которым важно качество и не страшно облако.
- Document context из #26 работает поверх любого провайдера (для DeepL — глоссарий через их API, topic/domain игнорируются; для Claude/Ollama — всё через prompt).

**Минусы.**
- Усложнение кода (~150 строк abstraction).
- Облачные провайдеры теряют нашу фишку с history N=3 + length hint (DeepL не принимает контекст; Claude принимает, но через другой API).
- Sync-семантика разная: некоторые провайдеры работают по batch, некоторые — поштучно.

**Файлы.** Новый пакет `src/translation/` с `base.py` (интерфейс) + `ollama.py` + `deepl.py` + `claude.py`. Старый `src/translate.py` — wrapper для backward compat.

**DoD.** В config один флаг переключает между ≥2 провайдерами. Тесты на каждый провайдер с моком. Реальный прогон на одном видео под двумя провайдерами — два разных финальных видео.

#### 28. Замена gemma:e4b на aya-expanse:8b (или другой specialized translator) — _из обсуждения локальных LLM_

> **Status (2026-05-03):** Проверено A/B-прогоном `gemma4:e4b` vs `aya-expanse:8b` на 1 видео (260 сегментов), ветка `experiment/aya-vs-gemma`, отчёт — `experiments/results/2026-05-03-aya-vs-gemma.md`. **aya-expanse:8b отклонён**: систематически фабрикует контент (отвечает на вопросы вместо перевода, добавляет выдуманный контекст, изобретает реплики), раскрывает аббревиатуры (удлиняет переводы → ухудшает atempo-синхрон), переводит proper names (`Anthropic` → `Антропике`). gemma4:e4b остаётся в `config.yaml`. Скрипт `experiments/aya_vs_gemma/run.py` сохранён как баркас для повторной A/B с другими кандидатами (qwen2.5:14b, vikhr-nemo:12b, gemma3:12b).

**Сложность: 5/100 · Эффект: ~~50~~ → 0/100** (после проверки aya; другие кандидаты не проверены)

**Контекст.** Текущая модель `gemma4:e4b` в `config.yaml` — generic chat-модель Google, не специализированная под перевод. Её качество en→ru — приемлемое, но это не топ среди моделей того же размера. **aya-expanse:8b** от Cohere — модель, специально обученная на 23 языках, среди которых русский — основной. Имеет высокую способность следовать инструкциям (важно для length hint и document context из #26), доступна в Ollama (`ollama pull aya-expanse:8b`). По сообществу и benchmarks — лучший локальный кандидат для en→ru на CPU/M-чипе.

**Альтернативы для A/B:**
- **`qwen2.5:14b`** или `qwen3:14b` — китайская семья, сильна на en/zh/ru, особенно крупные версии.
- **`vikhr-nemo:12b`** — Mistral-Nemo, дофайнтюненная под русский российским сообществом. Очень естественные формулировки.
- **`mistral-small:24b`** — универсал, но 24B на CPU медленно.
- **`gemma3:12b`** (в отличие от `gemma4:e4b`) — заметно лучше на переводе, но всё ещё не специализирована.

**Идея.** Pull нескольких моделей-кандидатов, прогон одного и того же 5-минутного отрывка через каждую, слепое сравнение русских переводов. Победителя — в `config.yaml`. Это работа на час, без изменений кода (только `model:` в config).

**Файлы.** `config.yaml` — поменять `translation.model`.

**Тонкость.** Размер модели влияет на скорость синтеза перевода. На 25-минутной лекции gemma:e4b прогоняется за ~минуты, qwen2.5:14b — заметно дольше. Если разница в качестве не радикальная — взвесить против скорости.

**Связь.** Это **самая дешёвая итерация** в сторону лучшего перевода. Делается до #1 / #2 / #3, потому что они улучшают **существующую** модель — а тут мы можем взять модель, которая **изначально** лучше, и тогда часть проблем (overshoot, неестественные формулировки) уйдёт сама.

**DoD.** A/B на 5 минутах видео: aya-expanse заметно лучше gemma по субъективной оценке (естественность + терминология). Если так — обновить `config.yaml`.

### A. Улучшение качества дубляжа

#### 29. Video-level setpts как четвёртый рычаг синхронизации — _AutoDub/tools/video_editing.py:102-115_

**Сложность: 25/100 · Эффект: 30/100**

**Контекст.** Все наши timing-механизмы (#5 cascading, #17 carry-over, #22 rubberband, #23 weighted silence) работают **на уровне аудио** — ускоряют TTS, перераспределяют тишину между сегментами. AutoDub добавляет четвёртый, ортогональный рычаг: если перевод не влезает в окно даже после стрейча — **слегка замедлить видео** под TTS через ffmpeg-фильтр `setpts={1/speed_ratio}*PTS`. На коротких сегментах ratio до 0.95× визуально незаметно (особенно на talking-head без быстрого движения), но даёт +5% бюджета длительности — этого часто хватает, чтобы остаться ниже опасного потолка стрейча и не задушить речь.

**Идея.** В `merge.py` добавить опциональный пасс: если после расчёта rubberband-ratio для сегмента он всё ещё выше порога (например 1.4× уже слышно даже на rubberband) — вместо упирания в cap снизить нагрузку, замедлив соответствующий участок видео через `setpts`. В первой версии — opt-in через config-флаг, потому что у нас нет детектора «безопасных» сегментов (без губ крупным планом, без динамичного движения).

**Связь с #5/#17/#22/#23.** Не заменяет, а дополняет: каскад из четырёх уровней — сначала carry-over (#17) → потом rubberband (#22) → если ratio всё ещё высокий, применить video setpts → совсем последний resort, atempo cap.

**Файлы.** `src/merge.py` — расширить timing-цикл, ffmpeg-граф для видео получает per-segment `setpts`. `config.yaml` — `merge.allow_video_slowdown: false` (по умолчанию выключено) + `merge.video_slowdown_min_ratio: 0.92` (нижний предел замедления).

**Тонкость.** Видео с быстрым движением (action, спорт, видеоклипы) на ratio < 0.95 даёт заметную «вязкость» — фильтр должен быть opt-in. Для основного usecase'а проекта (лекции, talks, интервью) — безопасно. Если когда-то появится автоматический детектор сцены — можно будет включать выборочно.

**DoD.** На сегменте, где русский TTS на ~50% длиннее английского — комбинация carry-over + rubberband + video slowdown укладывает дубляж без слышимой артефакции и без обрыва, при этом видео замедлено не ниже 0.92×.

### B. Оптимизация (не про качество)

#### 30. Параллельный TTS-синтез через worker pool — _AutoDub/tools/audio_synthesis.py:178-312_

**Сложность: 30/100 · Эффект: 0/100 на качество, 60/100 на скорость**

**Это оптимизация скорости пайплайна, не качества дубляжа.** Не путать с пунктами выше — финальный звук побитово (или субъективно при stochastic seed) совпадает с последовательным синтезом.

**Контекст.** Сейчас `src/tts.py` синтезирует сегменты последовательно — на 25-минутном видео это десятки секунд на CPU с Silero и потенциально минуты на тяжёлых cloning-моделях. AutoDub использует `multiprocessing.JoinableQueue` с N воркерами: пул процессов разбирает очередь сегментов, каждый воркер держит свою копию TTS-модели в памяти, итоговый speedup ~ N (минус накладные на загрузку модели в каждый процесс).

**Идея.** В `src/tts.py` обернуть batch-синтез в worker pool. Параметр `tts.workers: N` в config. Для Silero N=2-4 на типичной машине (модель ~60 MB на воркер). Для XTTS/F5 (#10) на GPU — N зависит от VRAM, обычно 1-2.

**Когда брать.** Сейчас — низкий приоритет (Silero и так быстрая, последовательный синтез не bottleneck). **После #10** (voice cloning) — становится почти обязательным: XTTS/F5 на CPU непрактичны без параллелизма, а на GPU позволяют утилизировать ресурсы.

**Файлы.** `src/tts.py` — обёртка над `synthesize_segment` через `multiprocessing.Pool` или `concurrent.futures.ProcessPoolExecutor`. `config.yaml` — `tts.workers: 1` (по умолчанию текущее поведение).

**Тонкость.**
- Загрузка Silero/XTTS в каждом воркере дублирует RAM. На XTTS v2 (~2 GB на инстанс) это критично — либо использовать shared model через очередь запросов вместо ProcessPool, либо ограничить `workers=1-2`.
- Логи из воркеров надо собирать через очередь, иначе в stdout перемешаются строки.
- Детерминизм результата сохраняется (каждый сегмент независим). На seed-зависимых моделях нужен фиксированный seed на воркер.

**DoD.** На 25-минутном видео с Silero и `workers=4` — TTS-фаза в 3-4 раза быстрее последовательной. Проверка идентичности: `ffmpeg -f null` хеши финального WAV совпадают с baseline (или субъективное сравнение, если модель stochastic).

## Эксперименты с TTS — voice cloning кандидаты

Расширение пункта #10. Ниже — конкретные модели для замены Silero, в порядке убывания приоритета для экспериментов. Цель — перейти от фиксированных 5 голосов Silero к **zero-shot voice cloning**: даём модели короткий референс-аудио (3-30 сек) оригинального диктора и текст перевода → получаем русскую речь его голосом. Это качественный скачок для дубляжа: зритель слышит «того же человека по-русски», а не озвучку.

Рекомендованный порядок: F5-TTS Russian → Fish Speech → XTTS-v2 → (откат) Silero. MMS-TTS — справочно как baseline.

**Тестовый набор для A/B.** Прежде чем сравнивать модели, заготовить набор фраз, которые ломают слабые TTS:
- Имена собственные (русские и иностранные).
- Числа и даты (`2026 год`, `15.4`, `три миллиона`).
- Аббревиатуры (`API`, `ML`, `НИИ`).
- Длинные предложения с придаточными.
- Вопросительные интонации.
- Ударения в омографах (`зАмок` vs `замОк`, `мУка` vs `мукА`).

#### F5-TTS Russian — приоритет 1

**Контекст.** F5-TTS — open-source модель из статьи «F5-TTS: A Fairytaler that Fakes Fluent and Faithful Speech with Flow Matching». В отличие от классических TTS (текст → mel → vocoder), использует **flow matching** — учится трансформировать шум в спектрограмму речи через ODE-траекторию, обусловленную текстом и референс-аудио. Существуют русские fine-tune'ы:
- **F5-TTS_RUSSIAN** — обучен на 5000 часах русской и английской речи. Карточка модели обещает поддержку явного управления ударениями в словах — критично для русского, где TTS на омографах обычно мажется.
- **hotstone228/F5-TTS-Russian** — альтернативный fine-tune, RU + EN, большой датасет.

Формат работы — zero-shot prompt-based: на вход 10-30 сек аудио оригинального голоса + текст → голос клонируется. Для дубляжа идеально вписывается: из оригинального видео уже извлекаем аудио, можно сразу нарезать референс на спикера (особенно после #15 diarization).

**Плюсы.** Поддержка ударений → решает главную боль русского TTS. Zero-shot cloning → голос меняется со сменой видео. Качество на русском — лучшее из доступного open-source.

**Минусы.** GPU практически обязателен (на CPU инференс непрактично долгий). Качество сильно зависит от качества референса (шум/реверберация в референсе → артефакты в выходе). Препроцессинг текста (нормализация чисел, аббревиатур, ударений) ложится на нас.

**Лицензия.** Базовый код F5-TTS — MIT. Русские fine-tune'ы: F5-TTS_RUSSIAN — CC-BY-NC-4.0 (некоммерческая), hotstone228 — CC-BY-NC-SA-4.0 (NC + ShareAlike). Для личных экспериментов и open-source проекта ок, для коммерческого продукта — блокер.

**Как пробовать.** Hugging Face репо `Misha24-10/F5-TTS_RUSSIAN` или `hotstone228/F5-TTS-Russian`. Установка через официальный репо F5-TTS, прогон CLI/Python API на нашем тестовом наборе. Сравнить с текущим Silero на одном видео, слепое прослушивание.

#### Fish Speech — кандидат на сравнение

**Контекст.** Fish Speech (от Fish Audio) позиционируется как multilingual SOTA open-source TTS. Архитектура — autoregressive transformer над neural audio codec токенами. Заявленный объём обучения варьируется в источниках: блог Fish Audio упоминает 720 тыс часов multilingual, более новые версии (Fish Audio S2 Pro) — 10+ млн часов и 80+ языков. Поддерживает voice cloning по референсу, instruction-tuning для эмоций.

**Плюсы.** Заявленное качество естественности и эмоциональной выразительности. Multilingual покрытие. Активно развивается.

**Минусы.** Multilingual ≠ хорошо на каждом языке. Для русского качество надо проверять — корпус мог быть смещён в сторону EN/ZH/JA. Не доверять self-claim «SOTA», только A/B на нашем тестовом наборе.

**Лицензия.** Проверять на момент использования — у Fish Audio несколько релизов с разными условиями (часть — open weights, часть — API-only, часть — NC).

**Как пробовать.** Github `fishaudio/fish-speech`. Прогнать тот же тестовый набор что и для F5-TTS, прямое сравнение. Особое внимание — ударения в омографах и числа.

#### XTTS-v2 — практичный baseline для voice cloning

**Контекст.** XTTS-v2 от Coqui — multilingual TTS с voice cloning по 3-секундному референсу. Не самая свежая модель (релиз ~2023), но проверенная: миллионы загрузок на Hugging Face, отлаженный API, обширное community. Использует GPT-style autoregressive backbone + HiFi-GAN vocoder.

**Плюсы.** Простое поднятие через Coqui TTS Python пакет — `tts.tts_to_file(text=..., speaker_wav=...)`. Стабильно работает на русском. Меньшие требования к GPU чем у F5/Fish. Зрелая документация.

**Минусы.** Не SOTA по качеству относительно F5/Fish. Управление ударениями ограниченное → на омографах плавает. Эмоциональная выразительность средняя.

**Лицензия.** Coqui Public Model License (CPML). После закрытия Coqui (2024) status коммерческого использования в community обсуждаемый — для личных/исследовательских экспериментов ок, для продукта — внимательно читать актуальную версию лицензии.

**Как пробовать.** `pip install TTS`, модель `tts_models/multilingual/multi-dataset/xtts_v2`. Запасной вариант, если F5-TTS_RUSSIAN не заведётся или окажется хуже на нашем наборе.

#### Silero TTS — текущий baseline

Уже используется (`v4_ru`). Описывать заново не нужно — см. CLAUDE.md и `src/tts.py`. Стоит держать в коде как fallback для CPU-only окружения и как стабильный референс для A/B сравнений новых моделей.

**Когда оставить Silero.** Простые лекции одного спикера на CPU без претензий на актёрскую выразительность. Дальнейшие эксперименты с voice cloning — поверх, через config-флаг `tts.engine`.

#### MMS-TTS Russian — справочно

**Контекст.** Чекпойнт `facebook/mms-tts-rus` от Meta из проекта Massively Multilingual Speech (1100+ языков). Доступен через `transformers`. Архитектура — VITS-based.

**Плюсы.** Простое поднятие через `transformers`. Работает на CPU.

**Минусы.** Один фиксированный голос, без cloning. Качество ниже Silero на русском по большинству оценок. Никакого управления ударениями.

**Вердикт.** Не стоит ставить выше F5/Fish/XTTS. Упомянут для полноты — на случай если интересна архитектурная альтернатива VITS для baseline-сравнений или потребуется покрытие редкого языка из MMS-семейства.

## Стриминг и live-режим

#### 31. Realtime-дубляж (генерация быстрее воспроизведения)

**Сложность: 90/100 · Эффект: 90/100**

**Контекст.** Сейчас пайплайн строго batch: пользователь скармливает файл/URL → ждёт N минут → получает `*_dubbed.mp4`. Альтернативная категория продукта — **realtime**: пользователь начинает смотреть, дубляж генерируется параллельно с воспроизведением и подаётся в плеер с латенси ≤2-3 сек. Это качественно другой UX и открывает новые сценарии (live-стримы, длинные видео, web-extension над YouTube).

**Уровни реализации (по возрастанию сложности).**

1. **Pseudo-realtime (offline-fast).** Самый дешёвый шаг. Пайплайн остаётся batch, но ускоряется до коэффициента **>1× реального времени** на типовом железе. Воспроизведение начинается, как только обработан первый минутный буфер; остальное «догоняется» параллельно. Архитектурно — почти то же, что сейчас, плюс progressive output (HLS-сегментация на лету) и плеер, который умеет читать растущий манифест. Главные enabler'ы у нас уже в backlog: #30 (параллельный TTS) + GPU-инференс для XTTS из #10 + #13 vocal separation вынесена в отдельный поток. Этого достаточно, чтобы первая версия realtime работала на записанных файлах.

2. **True streaming (сегмент-за-сегментом, без файла целиком).** Пайплайн перестроен в pipe: ASR работает на скользящем окне аудио (`whisper-streaming`, `WhisperLive`, или native faster-whisper streaming API), сегменты по мере появления уходят в MT, оттуда в TTS, оттуда в playback queue. Каждый этап — независимая корутина/процесс с очередью между ними. Latency end-to-end складывается из ~1с ASR window + ~0.5с MT + ~0.5-1с TTS = 2-3 сек. Фундаментальное ограничение: **streaming MT теряет контекст N=3** из #2 (мы переводим текущий сегмент, не зная следующих) — качество перевода падает, особенно на длинных предложениях.

3. **Live source (трансляции).** Не записанное видео, а Twitch/YouTube Live. Все ограничения уровня 2 + жёсткий бюджет latency (зритель не ждёт 30 сек) + ASR должен справляться с шумным live-аудио без второго прохода. Lip-sync (#21) на live невозможен в принципе. Реалистичное достижимое — overdub в стиле «синхронного перевода ООН» с ощутимой задержкой.

**Идея для первого приближения.** Делать только уровень 1 (pseudo-realtime). Это требует:
- Декомпозиция `dub.py` из последовательного `transcribe → translate → tts → merge` в **граф из независимых стадий с очередями** (например `asyncio.Queue` или `multiprocessing.Queue`). Каждая стадия — отдельный воркер, читает из своей входной очереди, пишет в выходную.
- Готовые сегменты сразу собираются в **HLS-фрагменты** (mpegts по 6-10 сек) и публикуются в манифест `playlist.m3u8`. Любой HLS-плеер (`hls.js` в браузере, ffplay, VLC) может начать воспроизведение, не дожидаясь конца.
- Метрика: на 25-минутном видео первый кадр должен выйти на экран **через ≤30 сек** после старта (а не через 5-10 минут как сейчас).

**Зависимости.** Опирается на:
- **#30 параллельный TTS** — без него bottleneck в синтезе, realtime не достижим даже на быстром железе.
- **#10 voice cloning на GPU** — XTTS v2 на GPU укладывается в 0.2-0.3× realtime; на CPU исключено.
- **#12 Segment metadata persistence** — манифест нужен для координации стадий.
- Опционально #13 (vocal separation) — выносится в отдельный воркер, чтобы не блокировать остальной граф.

**Идея вынести из браузера.** Пользователь правильно отметил: **обычный браузер не подходит** для серьёзного realtime-дубляжа поверх стороннего видео.
- **DRM-видео (Netflix, Prime, YouTube Premium с DRM-треками) — недоступно в принципе.** Web-плеер не отдаёт сырое аудио в JS; вытащить его на сервер для ASR нельзя.
- **Заглушение оригинальной дорожки + наложение синтеза.** В браузере доступно через Web Audio API (MediaElementAudioSourceNode + GainNode), но требует CORS-разрешений на источник. На YouTube/embedded-видео работает не везде.
- **Latency сетевой доставки.** Стриминг WAV-фрагментов через WebSocket добавляет ~200-500 мс к pipeline-latency. Для UX «синхронного перевода» это много.
- **Альтернативы:**
  - **Native desktop-приложение** (Tauri / Electron + Python backend). Свой mini-плеер, полный контроль над аудио-микшером, прямой доступ к локальным файлам без upload.
  - **Browser extension** (Chrome MV3) для не-DRM источников: расширение перехватывает HTML5 `<video>`, отправляет аудио на локальный backend, получает дубляж через локальный WebSocket. Работает на YouTube/Vimeo/собственных сайтах.
  - **Reverse proxy / OBS plugin** для стриминговых сценариев: вмешательство в видеопоток до того, как он попадёт в плеер.

Я бы начинал с **native-приложения для локальных файлов** как минимально жизнеспособной realtime-сборки — никакого DRM, никакого CORS, можно фокусироваться на самой пайплайн-механике. Browser extension и live-источники — следующие итерации.

**Файлы.** Это, по сути, новый проект поверх существующего ядра:
- `src/pipeline/streaming.py` — оркестратор графа стадий с очередями.
- `src/server/hls.py` — публикация HLS-манифеста и mpegts-фрагментов.
- `src/server/api.py` — управляющий API (старт/стоп задания, прогресс, доставка фрагментов).
- Опционально `client/` — Tauri-приложение или browser-extension как UI.
- `src/transcribe.py`, `src/translate.py`, `src/tts.py` — рефакторинг под streaming-семантику (вход — async iterator вместо list).

**Тонкость про качество.** Streaming-режим **снижает качество** перевода и сегментации:
- Streaming ASR хуже batch Whisper (нет финального прохода для коррекции).
- Streaming MT теряет контекст N=3 (см. #2).
- Cascading-playback (#5), carry-over silence (#17), weighted silence (#23) частично работают только в batch — в streaming мы не знаем, сколько у нас «slack» впереди.
- Поэтому realtime-режим должен быть **отдельным флагом**, не заменой batch. Кому нужно качество — оставляет batch; кому нужно «здесь и сейчас» — переключает.

**DoD.** На записанном 25-минутном видео с GPU и `--realtime`: первая секунда дубляжа звучит ≤30 сек после запуска, общая обработка идёт быстрее воспроизведения (генерация опережает плеер с положительным запасом до конца видео), синхронность голос/видео не хуже batch-варианта на ту же модель.
