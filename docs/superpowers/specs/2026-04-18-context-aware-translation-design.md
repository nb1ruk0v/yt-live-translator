# Context-Aware Translation — Design

## Проблема

Сейчас `translate.py` переводит каждый Whisper-сегмент изолированно, через `/api/generate`, с минималистичным промптом и без параметров сэмплирования. Это порождает три класса дефектов качества:

1. **Потеря связности.** Местоимения (he/she/they/it), род существительных, перевод терминов выбираются наугад, поскольку LLM не видит предыдущих реплик.
2. **Мусор в ответе.** llama3.1:8b периодически игнорирует инструкцию «Return only the translation» и добавляет префиксы (`Here's the translation:\n...`), кавычки, пояснения. Этот мусор уходит прямо в TTS.
3. **Недетерминированность.** Без `temperature: 0` перевод одного и того же видео от запуска к запуску отличается, что мешает отладке.

## Решение

Переписать `translate.py`, сохранив публичную сигнатуру:

```python
def translate(segments: list[Segment], config: dict) -> list[Segment]
```

### Ключевые изменения

1. **Перейти с `/api/generate` на `/api/chat`** — нативная поддержка history.
2. **System prompt** фиксирует роль и формат вывода.
3. **Sliding window контекста N=3** последних пар `(оригинал, перевод)` подаётся как диалоговая история. Значение выбрано как компромисс между связностью и рисками (эрозия релевантности, распространение ошибок перевода по истории, латентность без prompt-кэша в Ollama).
4. **`options.temperature = 0`** — детерминированный перевод.
5. **Постобработка ответа:**
   - trim пробелов и кавычек (`"`, `'`, `«`, `»`);
   - при многострочном ответе брать первую строку с кириллицей (отсекает префиксы типа `Here's the translation:`);
   - валидация: непустой и содержит хотя бы один кириллический символ.
6. **Фолбэки:**
   - Пустой `seg.original` → `seg.translated = ""`, без обращения к Ollama.
   - Невалидный ответ LLM → `seg.translated = seg.original` + предупреждение в stderr. Цель — не ронять пайплайн после дорогой транскрипции.
7. **Таймаут** поднять с 60 до 120 секунд: chat c историей обрабатывается медленнее.

### Что НЕ меняется

- `config.yaml` — без новых ключей. `N=3` хардкодится как константа модуля; вынесем в конфиг, если появится необходимость тюнить.
- `Segment`, `transcribe.py`, `tts.py`, `merge.py`, `dub.py` — без изменений.

## Архитектура модуля

```
translate(segments, config)
    │
    ▼
  для каждого seg:
    ├── empty? → skip
    ├── build messages: [system] + history[-3:] + [user: seg.original]
    ├── POST /api/chat (temperature=0)
    ├── clean(response)
    ├── validate (кириллица есть) → иначе fallback
    ├── seg.translated = result
    └── history.append((seg.original, seg.translated))
```

`history` — локальная переменная функции, не переживает вызов. Между запусками пайплайна состояние не сохраняется.

## Тестирование

- **Smoke-test:** полный прогон `uv run dub.py data/test_video.mp4`. Проверяем, что пайплайн отрабатывает без ошибок и итоговый MP4 создан.
- **Качественная оценка:** глазами сравниваем переводы до/после на том же видео — особенно в сегментах с местоимениями и продолжающимися фразами.
- **Формальных unit-тестов нет:** перевод — это вызов к внешней недетерминированной (даже при temperature=0 зависит от модели/версии Ollama) LLM, стабильно ассертить бесполезно. Постобработку (`_clean`) можно покрыть unit-тестами, если понадобится регрессионная защита.

## Риски

- **Контекст-окно Ollama по умолчанию 2048 токенов** (а не 128k модели). При длинных сегментах + N=3 можно случайно упереться и получить молчаливое обрезание. На практике субтитры короткие — не должно мешать, но если симптомы появятся, выставим `options.num_ctx: 8192`.
- **Ошибка в одном переводе отравляет следующие N=3** вызова через history. Фолбэк на оригинал при невалидном ответе частично смягчает (в history тогда попадёт английский текст — модели видно, что это «не норма»).

## Out of scope

- Пункты #2 (VAD в transcribe) и #3 (atempo в merge) из ревью — отдельными задачами.
- Вариант B (sentence-grouping + redistribution) — отдельной задачей, если N=3 окажется недостаточно.
- Кэширование переводов между запусками.
